In this sprint, we will go to zoom into the undefined entity Status that will contains everything is needed to maintain the state of the system.

Introduction

From the analysis, it is clear that the main server that realizes the application logic must have a way to maintain the state of the entire system. In order to do this, in this sprint we are going to project a set of persistence classes that can be used by the APP-SERVER. To meet the requirements it is not necessary to have a complex persistence structure because all data to maintain are:
  • the state of the doors (indoor and outdoor);
  • the user that are using the system;
  • the state of all parking slot.
At this moment, the consumer requires to have only six slot and, in addition to this, he has no claims about the registration of the user. So, in this first developement, we decide to use shared variable to save the state. Notice that it is strongly discouraged because of if the system takes demage then it can loss all data, but it is the fastest method to easily have a working prototipe. Then, if the classes will be designed with strong extensibility, so will be easy to save the state on another better support.

Bean classes

Package it.unibo.parkmanagerservice.bean

The model of the entire application is composed of Java Bean, which are Java classes with a particular convention that allow to map persistent entities of the system in the Object World.
Since we are using QAK-System, and then Kotlin, instead of the classic Java Beans, we will use the Kotlin Data Classes which are specific for persistence. These classes will be easily reusable in future developements also with the use with a database (see the tutorial Building web applications with Spring Boot and Kotlin).

The two domain entities that require a Kotlin Data Class:
  • User: User.kt
  • Slot: ParkingSlot.kt
In addition to this two entities, also the doors state deserves a data class:
  • Door State: LocalDoorState.kt
The UML diagram of the package it.unibo.parkmanagerservice.bean is:
User.kt
This data class maintains the information about an user. Every user must have:
  • an id that must be univoque and generated by the system;
  • a name and a surname that can be directly given by the user;
  • a mail that must be univoque and given by the user;
  • a state that can be modified while the user is in the system.
About timestamp, this property is inserted because of the user can be inserted into some queue, so it is very useful to calculate the queue time for the relative user. It also allow to realize policies for entering and exiting the queue. This attribute is not needed if the state is maintained into local variables but is necessary for use with a database or other complex structure that have no built in queue.
UserState.kt
This enum class represents the state that a user can have:
  • CREATED: the user has correctly been created and inserter into the data structure;
  • INTERESTED: the user has notified the interest to park car, but the indoor is not free so he is waiting for it;
  • INDOOR_RESERVED: the user has obtained the access to the indoor that has been reserved for him;
  • PARKED: the car of the user has correctly been parked into the parking area;
  • WANTS_TO_GO: the user has notified that he wants to pick up his car but the outdoor is not free so he is waiting for it;
  • OUTDOOR_RESERVER: the user has obtained the access to the outdoor that has been reserved for him;
  • PICKED_UP: the user has picked up his car and exited the system; this state has been inserted only for future development, if the consumer requires a sort of registration for users or he is interested to maintain some datas about users.
ParkingSlot.kt
This class maintains the information about a parking slot. Every slot must have:
  • a slotnum that is univoque and cannot change during the execution of the server application;
  • a slotstate that can evolve.
In addition, this class map the One To One relationship between users and parking slot: indeed, a slot can be reserved or used by a certain user so the property user let the system to get who is the user releated to the slot. If no user is using the slot, then this property must be setted to null. The user property can also be used to retrieve the slot by a given user token (TOKEN) simply by checking 
slot.user.token == TOKEN
ParkingSlotState.kt
This enum class represents the state that a slot can have:
  • FREE: the slot is free and can be reserved for an user;
  • OCCUPIED: the slot is occupied by a car of an user;
  • ALMOST_FREE: the user associated with the slot has requested to pick up the car, but it still is on the slot;
  • RESERVED: the slot is reserved for a certain user.
Notice that in the states RESERVED or OCCUPIED, the slot must have setted the slot.user property with the user that is using the slot. Otherwise, for the FREE state, the property must be null.
LocalDoorState.kt
This class maintains the informations about the two door:
  • the state of the indoor with the user that is using it, that is null if the door is free;
  • the state of the outdoor with the user that is using it, that is null if the door is free;
Notice that this class implements DoorsManager in order to be more reusable in the hypotesis to use a database.
DoorState.kt
This enum class represents the state that a door can have:
  • FREE: the door is free and can be used by the system;
  • OCCUPIED: the door is occupied by the car of a user that is over it;
  • RESERVED: the door is reserved to an user so the car is going to be on it.
As said, in the state RESERVED or OCCUPIED, the user related to the door must not be null and specularly, with the FREE state, the user should be null.
DoorsManager.kt
This interface defines the behavior of a Door Manager then has a lots of utility methods to set the state of a door:
  • getState() and setState() are the getter and setter for the state of the port;
  • getUserAtDoor() and setUserAtDoor() are the getter and setter for the user related to the doors;
  • reserveForUser() is the utility method to reserve a door to a certain user;
  • setFreeWithNoUser() is the utility method to free a door and the user associated to it.
The operation that that system can perform using manager are: 
val USER = ...
//...
manager.reserveForUser(DOOR, USER)
//The door DOOR is now reserved to the user USER
//...
manager.setState(DOOR, LocalDoorState.OCCUPIED)
//The door DOOR is now occupied by the user USER
//...
manager.setFreeWithNoUser(DOOR)
//The door DOOR has been set to free and relative
//user is null

Persistence classes

Package it.unibo.parkmanagerservice.persistence

The class in this package realize the concept of repository; so, every class of the bean package, must have an own repository that implements the CRUD operations over the data structure.
UserRepository.kt
This interface defines the behavior of a repository for the bean User. Besides the CRUD methods, it defines some useful operation to retrieve the users with a certain condition (by the state, by the mail or by token). These methods return null or Optional.empty if no user is found. Notice the method getFirstInState() that must search the first user that is transited to the passed state: this method can be used from another entity in order to realize queues of priority.
BasicUserRepository.kt
The implementation of UserRepository using local memory. All users are saved into a Map. Notice that the method create() also set the id of the user using the idgen generator. In addition, this method could throw an SQLException if the user to store has a mail already used.
ParkingSlotRepository.kt
This interdace defines the behavior of a repository for the bean ParkingSlot. Besides the CRUD methods, it defines some useful operation to retrieve the slots with a certain condition (by the state, by the token of the relative user, by the relative user the slot is reserved or used for). In order to obtain the first free slot, the system can invoke the method getFirstFree() on the implementation object.
BasicParkingSlotRepository.kt
The implementation of ParkingSlotRepository using local memory. All slots are saved into a Map.
DoorQueue.kt
This interface is an abstraction for realizing the concept of door queue. Infact, it is very frequently that more that one user is competing for accessing a door. The operations exposed by this interface are the classic ones of a queue:
  • the insertion of an element (addUser() method);
  • the extraction of an element from the queue (pollNextUser() method);
  • the get for the current length of the queue (remaining() method)
This interface could also be implemented in order to use a database to realize queue.
LocalDoorQueue.kt
The implementation of the interface DoorQueue that use a Kotlin ArrayDeque.
DoorQueues.kt
An utility class that is sort of double singleton. This class is a point of global access for the two door queues of the system, then istantiatiating and maintaining the unique queues associated to the doors. In order to access the queue, the system must perform: 
DoorQueues.getIndoorQueue()
//Return the indoor queue

DoorQueues.getOutdoorQueue()
//Return the outdoor queue
ParkingRepositories.kt
An utility class that is sort of double singleton. This class is a point of global access for the two repositories of the system (user and slots), then istantiatiating and maintaining them. In order to acces the repositories, the system must perform: 
ParkingRepositories.createBasics()
//Create the two repositories in their basic version

ParkingRepositories.getUserRepository()
//Return the unique user repository

ParkingRepositories.getParkingSlotRepository()
//Return the unique slot repository


Pay attention beacause these implementation of repository can not be used in a concurrent environment, so it is not safe to access repositories bot in writing and reading way from more than one actor. If you want to share them with more than one actor, so there are needed others coroutine-safe implementations.
In addition, in a future development, it is possible to set the system to use a database or another data structure by:
  • implementing a new classes for UserRepository, ParkingSlotRepository and LocalDoorQueue;
  • modify the methods contained into ParkingRepositories and DoorQueues in order to use the new classes.
