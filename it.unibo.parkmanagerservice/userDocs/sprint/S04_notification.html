In this sprint, we will go to zoom into the notification system in order to meet the requirement of notification to the user and the manager more efficienly.

Introduction

In the two analysis, it is only said that client and manager must be notificated in their GUI. However, if the GUI are a web or mobile application, it is very likely that the user or the manager close the view or the application: anyway, the system must have a way to send asynchronous and non-volatile notification regardless of the GUI, such as an SMS or a mail.
From the analysis, the main situation that require a strong notification are:
  • when the user does not move the car at the indoor after ITOCC is reached;
  • when the user does not take out the car from the outdoor afret DTFREE is reached;
  • when DTFREE is reached, the manager must be notified.
In addition to these situation, that came from the requirement, it would be good if:
  • the client is notified with the slotnum;
  • the client is notified with the token;
  • the client is notified when the car is into the outdoor.
We decide to use a notification based on the mail. Then, when the system need, it has the possibility to simply send a mail to the interested user.
Like the previous sprint, also this component must be designed in such way that it is easily extendable.

Notification Layer

Package it.unibo.parkmanagerservice.notification

In order to send notification via mail in Java, we will use the library JavaMail. Obviously, it is necessary to have an email provider with a valid account and configure the JavaMail API with the SMTP or POP configuration given by the provider.
Notification.kt
This data class represents a notification. It contains all attribute that all notification should have (such as destination, content, subject, ecc...).
NotificationType.kt
This enumeration defines all possible type of notification intended as the situation that requires it. Then:
  • SLOTNUM: notification that will be sent that contains the slotnum reserved to the user;
  • TOKEN: notification that contains the token of the user and that will be sent after he has moved the car to the indoor;
  • PICKUP: notification that will be sent when the user car is positioned at the outdoor then the user can take it;
  • LOSEN_RIGHT: notification that will be sent if the user has losen the right to park because of ITOCC is reached;
  • ADMIN_DTFREE_REACHED: notification that will be sent to the manager when a DTFREE is reached;
  • USER_DTFREE_REACHED: notification that will be sent to the user when his DTFREE is reached and he has not taken his car.
Notifier.kt and FakeNotifier.kt
The Notifier interface defines the behavior of a component dedicated to send notification. The FakeNotifier implements this interface using an empty sendNotification() method that only print the notification to the standard output.
MailNotifier.kt
An implementation of Notifier interface that use JavaMail to send notification via mail. This component is designed in order to be configurable with a file located at configs/mail.json that contains the information about the SMPT connection to the provider (such as username, password, host, ecc...). The constructor of this object read the first line of this configuration that is a JSON entry with the format: 
{"mail":"xxx@xxx.xxx", "pass":"xxxx", "host":"xxx.xxx.xx", "port": "00", "ssl-enabled": "true/false", "auth": "true/false"}
DefaultNotificationFactory.kt
This Factory Interface allow to easily create notification with default content. In the next section we will describe how the content is formatted but for now we specified that this class has a configuration file located at configs/notification.json
SystemNotifier.kt
A global point access for the unique Notifier of the system. It build the Notifier using a configuration file located at configs/notifier.json formatted as: {"type":"mail"} At the moment only two value are allowed for the key type: fake and mail, relative to the two implementation of the Notifier interface.


Then, these POJO will be used by a dedicated actor that can be inserted into the same context of the APP-SERVER:
NotificationChannel.kt
In order to correctly user this mechanism, it is need to pass the notification to send to the notifieractor. To do this, there are several possibilities:
  • defining a message notifyuser with all arguments of the notification; then the actor create the Notification object using the factory and send it with the Notifier object; ⇒ if the content of the notification is too long, so the message can be very heavy
  • building a notification from the actor who wants to send it and serialize in order to encapsulate it in the message notifyuser that, in this case, as only one argument (the serialized notification) ⇒ the serialization could be a little bit difficult and the format of the QAK message can prevent the sending
  • building a notification from the actor who waints to send it and pass it to the notifieractor by using shared memory or a structure embedded in Java/Kotlin.
Then, we have choosen to use the last possibility, so we have defined the class NotificationChannel.kt that is a singleton for accessing a channel in which are putted the notification to send: the notifieractor read the notification from the channel and send it using the system notifier. This choice makes the notifieractor usable only in the context he is, but is very efficient and easily implementable. In fact, every QAK actor is a coroutine, so it is strongly advised to use the Kotlin structure for coroutine. So, if a QAK actor in the same context of the notifier wants to send a notification, could use: 
[# 
	NOTIFICATION = DefaultNotificationFactory.createForUser(...) 
	NotificationChannel.CHANNEL.send(NOTIFICATION)
#]
	forward notificationactor -m notifyuser : notifyuser(NOTIFY)


Summing up, we have introduced a new entity, the notifieractor, that is a QAK actor able to receive the request to notify a certain user. In this way, the other actors can delegate the notifications to this entity and can not block themself to send notification. Notice that is a good practise first to write the notification inside the channel (such as in the previous one code), and only after to send the notifyuser messagge to the notifier actor.

Mail format and customization

In order to customize the mail, a great graphic has created a logo for the application:
This logo has been used into all notification mail in order to customize the layout of them.
To customize the mail layout, we choose to use HTML Mail. So, we have created a template for all notification type:
  • SLOTNUM: slotnum_mail.html (see preview)
  • TOKEN: token_mail.html (see preview)
  • PICKUP: pickup_mail.html (see preview)
  • LOSEN_RIGHT: losen_right.html (see preview)
  • ADMIN_DTFREE_REACHED: admin_dtfree_reached.html (see preview)
  • USER_DTFREE_REACHED: user_dtfree_reached.html (see preview)
  • GENERAL: general.html (see preview)
    • These preview are used by the DefaultNotificationFactory in order to create a sort of notification template. Indeed, these html contains some field that start with '$': the factory replaces this fields with the correct values depending on the notification type (see the method createForUser() of DefaultNotificationFactory.kt for more details).
    In addition to this, in general every notification has an html with a logo and a button inside. In order to make this layer more flexible, is designed a configuration file in JSON used by the factory at the location configs/notification.json. An entry of this file is similar to: 
    {"type":"SLOTNUM", "subject":"Invio SLOTNUM [ParkManagerService]", "content-type": "text/html", "html":"resources/slotnum_mail.html", "linkpattern": "http://localhost:8080/carenter?slotnum=$SLOTNUM&usermail=$USERMAIL"}
    This line configure the SLOTNUM notification with the subject, the content-type (at the moment only text or html/text are allowed), the html location, and the pattern for the link associated to the button. In addition to this line (one for any notification type), this configuration file has also the two lines: 
    {"button":"resources/button.html"}
{"logo":"resources/logo.html"}
    that respectly indicate the location of the html of the button and the path of the logo file.

Testing

The test of the mail notification system is not easy and is hardly automatizable (but not impossible). However, in order to do a simple and a fast test, it is possible to send test mail of all type to a real address, checking if the mail are received. Then, the pattern for this simple test is: 
let TYPES = < array containing all NotificationType enum >
let USER =  < test user with real existing mail >
for T in TYPES do
	let NOTIFICATION = < create T notification from DefaultNotificationFactory  >
	< send NOTIFICATION using MailNotifier  >
end

TestMailNotifier.java

After the execution of this test, the received mail are (in Outlook format):
  • slotnum.msg
  • token.msg
  • pickup.msg
  • losen_right.msg
  • admin_dtfree.msg
  • user_dtfree.msg
  • general.msg
