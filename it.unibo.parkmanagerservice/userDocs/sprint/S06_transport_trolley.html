In this sprint we are going to design the trolley actor in order to realize the business logic operation on the physical robot. As specified in the analysis, the consumer give us the code to send the command to the physical robot, the basicrobot, that can manage three type of robot: virtual, nano or mbot. The main documentation about this software is on consumer github repository: basicrobot2021. Then, we are going to use this actor with another high-level actor that allow the system to easily send the robot the command to park or pickup a car. In addition to this, as required, there must be the possibility of stop and resume the trolley.

Introduction

We have already said that the trolley is mapped in the APP-SERVER as an actor. So, from the analysis, the trolley must expose these operations:
  • park a car to a certain slotnum;
  • pick up a car from a certain slotnum;
  • stop himself (with a pause logic);
  • resume himself after a stop.
We also specify that these are high-level operation that corresponds to a certain set of the low-level operations defined into the analysis:
  • park = moveToIn + moveToSlotIn + backToHome;
  • pick up = moveToSlotOut + moveToOut + moveToHome.
In addition to this, it is needed to implements a mechanism to maintain the state of the transport trolley (IDLE, WORKING or STOPPED). This state will be observable as the previous sprints, then using CoAP.

Utility classes

Package it.unibo.parkmanagerservice.trolley

SlotMap.kt
This class is an utility class that contains the allowed position for the robot associated to the slotnums. So, when the robot has to go to a certain slotnum to load or unload a car, the trolley can ask to the SlotMap to get the position to go near the slot. This mechanism is configuranle by a JSON file located at configs/slots.json that as lots of this entry, one for slot present at the parking area: 
{"slotnum": "1", "x":"3", "y":"1", "allowedX":"4", "allowedY":"1", "orientation":"right"}
  • slotnum: the slotnum to configure;
  • x and y: the coordinates of the slot;
  • allowedX, allowedY: the adiacent position near the slot used by the robot to load/unload the car;
  • direction: the direction the robot must have to load/unload the car.
DoorMap.kt
This class is an utility class that equivalent of the preious but for the two doors. It also has a JSON file located at configs/doorsmap.json that as two entry, one for door: 
{"door": "INDOOR", "x":"6", "y":"0", "allowedX":"6", "allowedY":"1", "direction":"downDir"}
  • door: the door to configure (indoor or outdoor);
  • x and y: the coordinates of the door;
  • allowedX, allowedY: the adiacent position near the door used by the robot to load/unload the car;
  • direction: the direction the robot must have to load/unload the car.
TripStageType.kt and TripStage.kt
The class TripStage defines a stage of the complete trip of the robot. It contains the destination of the current stage and also the type, that corresponds to those defined in the analysis and explained at the introduction.
The only difference between TripStageType and the operations defined into the analysis is that in this enumeration there is no difference between moveToSlotIn and moveToSlotOut: this two are collapsed in TripStageType.MOVING_TO_SLOT. Same for backToHome and moveToHome, that are collapsed into TripStageType.MOVING_TO_HOME. Notice that in the trip stage with type LOAD_CAR and UNLOAD_CAR, the trolley has not to move and stays in the same position, so the destination must be ignored.
CompleteTripBuilder.kt
A simple Builder Class that easily allow to create a complete trip. For complete trip we refer to the stage related to the high-level trolley operation (park and pick up).
In order to create a CompleteTripBuilder object, are need the coordinates of the indoor and the outdoor: 
val builder = ControllerBuilder(< INDOOR_X >, < INDOOR_Y >, < OUTDOOR_X >, < OUTDOOR_Y >)
So, to build the default complete trip, the trolley should use:
  • park: 
    vat tripIterator = builder
    .clean()
    .addParkTrip(< SLOT_X >, < SLOT_Y >)
    .build()
  • pick up: 
    vat tripIterator = builder
    .clean()
    .addPickupTrip(< SLOT_X >, < SLOT_Y >)
    .build()
MapLoader.kt
This class is an utility class that let the system to easily load the RoomMap from a txt file. Indeed, the consumer does not give us the .bin file required to load the map using the default loader, but only a .txt containing the map (parkingMap.txt). For details about RoomMap and Planner, see the document and the code given by the consumer at LabPlanner.html.
TrolleyState.kt
This enum simply represents the states in which the trolley can be as defined in the requirements:
  • IDLE;
  • WORKING;
  • STOPPED.

Trolley as actor

General view

The structure of this actor is a little bit complicated, but it is designed to be as reusable as possible. Indeed, the main logic of this actor is:
  1. find the position of the destination, by parsing the received dispatch (pick up or park);
  2. if the position is valid, calculate the complete trip, using the CompleteTripBuilder;
  3. iterate over all stage;
  4. for all stage, calculate the robot actions by using the plannerUtil;
  5. iterate over the actions and send the command to basicrobot for executing it.
In addition to this logic, the actor must save his state using CoAP. This state consists in a JSON string that has this format: 
 {"state":"WORKING","action":"MOVING_TO_SLOT","position":{"x":"6","y":"2"}}
  • state: is the state of the trolley (WORKING, IDLE, or STOPPED);
  • action: is the action that the robot is performing (a value of TripStageType) or null if the robot is in IDLE state;
  • position: a JSON object that contains the coordinates of the robot into the map.

States description

Considering that the trolley actor has a complicated structure, it is needed a small description of all states:
  • home: in this state the robot is at home and the state of the trolley is IDLE.
  • handle: in this state the trolley handles the dispatch sended by the other entities and calculate the stages he should perform. The slotnum arrived from the dispatch is parsed as coordinates stored into the variable DEST: then, if no slot was found, DEST is null and the trolley return in the state home.
  • nextStage: in this state the trolley makes an iteration on the stage iterator. So, if the iterator has no more elements, the boolean CANGO is set to false (the trolley has ended the trip, then return to home). Otherwise, if another stage is present, the robot go to the state checkStage. In addition to this, if the current stage contains some robot moves to do, in this state the robot use the plannerUtil to calculate them (but if no action was found, then CANGO is set to false (an error is thrown).
  • checkStage: this state has the only role to route the current stage by its type (if is load/unload car then go to elevator, otherwise is an action, then go to nextAction.
  • elevator: this state manages the stages that require to load or unload a car. In this state, the trolley corrects the direction of the robot in order to allow it to load or unload the car, then execute the operation.
  • nextAction: in this state the trolley get the next action to do from the iterator obtained by the plannerUtil. If the current action is null or is noOp, than CANGO is set to false and the trolley goes to analyze the next stage.
  • doAction: in this state the trolley request to the basicrobot to execute the current action to do. If the action requires a step message (the action is a forward), than CANGO is set to true and the trolley goes to wait the response, otherwise it sends the cmd message, and goes to obtain the next action. Notice that after a cmd dispatch sent it is needed to insert a small delay, in order to wait that the robot has executed the action (that certainly is a turn).
  • waitStepDone: in this state the trolley wait for a stepdone response after its step request. Notice that in this state it is possible to stop the state of the robot.
  • handleStepDone: this state simply handles the received stepdone and go to nextAction
  • stopped: in this state the robot is stopped. Notice that if this state is reached from waitStepDone it is needed to handle the stepdone response.
Also pay attention to the INTERRUPTIBLE variable, that can make a certain stage interruptible: in other words, if this value is true, then the trolley stop its stage and serve a new request. This variable becomes true only if the stage is MOVING_TO_HOME type.
As a final consideration, we decide that the trolley can be also stopped in the states home and waitStepDone, so only if:
  • the robot is at home, in the state IDLE;
  • the robot is waiting for a stepdone, then if the robot is walking, in the state WORKING.

In addition to all this, we leave some open points:
  • what if a stepfail is received?
  • how is possible to load and unload the car? What device is required?

Testing

The test of the trolley consists in a serveral situation in which the trolley may find itself. However, the main patter for all test case is: 
let HISTORY =  < empty String list >
let OBS = < istantiate CoAP observer >
OBS.start()
< start trolley actor >

while ((OBS.nextChange().state) != "IDLE")

forward < park/pickup dispatch > to trolley

while ((OBS.nextChange().state != "WORKING")

while((let CHANGE = OBS.nextChange()).state != IDLE) {
    add CHANGE to HISTORY
}

< check HISTORY contains the expected values >
