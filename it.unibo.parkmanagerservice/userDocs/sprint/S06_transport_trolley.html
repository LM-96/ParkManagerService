In this sprint we are going to design the trolley actor in order to realize the business logic operation on the physical robot. As specified in the analysis, the consumer give us the code to send the command to the physical robot, the basicrobot, that can manage three type of robot: virtual, nano or mbot. The main documentation about this software is on consumer github repository: basicrobot2021. Then, we are going to use this actor with another high-level actor that allow the system to easily send the robot the command to park or pickup a car. In addition to this, as required, there must be the possibility of stop and resume the trolley.

Introduction

We have already said that the trolley is mapped in the APP-SERVER as an actor. So, from the analysis, the trolley must expose these operations:
  • park a car to a certain slotnum;
  • pick up a car from a certain slotnum;
  • stop himself (with a pause logic);
  • resume himself after a stop.
We also specify that these are high-level operation that corresponds to a certain set of the low-level operations defined into the analysis:
  • park = moveToIn + moveToSlotIn + backToHome;
  • pick up = moveToSlotOut + moveToOut + moveToHome.
In addition to this, it is needed to implements a mechanism to maintain the state of the transport trolley (IDLE, WORKING or STOPPED). This state will be observable as the previous sprints, then using CoAP.

Utility classes

Package it.unibo.parkmanagerservice.trolley

General view of the controller package

ParkManagerServiceController.kt and KParkManagerServiceController.kt
The ParkManagerServiceController is the main interface that defines the methods that the controller implementation must have in order to realize the business logic. This interface has only one implementation, KParkManagerServiceController, that uses the classes seen in the previous sprints (bean, repository, doors management). All methods of this classes will be discussed soon.
ErrorType (inside ParkManagerError.kt)
This enumeration defines a possible error returned by some method of the controller. Then:
  • INVALID_SLOTNUM, INVALID_TOKEN, INVALID_MAIL: errors thrown when slotnum ot token or mail are not valid;
  • NO_USER_AT_DOOR: errors throws when there is no user at the door (but he is expected by the caller);
  • DOOR_NOT_RESERVED: error returned when the door is not reserved (but it is expected by the caller);
  • NO_RESERVATION_FOR_USER: error throws when there is no reservation for the user passed as argument;
  • NO_NAME, NO_SURNAME, NO_MAIL: error returned when is missing some data;
ControllerBuilder.kt
A simple Builder class in order to build and get the controller of the system. Obviously, this controller is also a Singleton then, only one creation is permitted: if the controller has been already built, then the invocation of the methods createK() or inject() has no effect. To build and get the controller using basic repositories and local door state, the APP-SERVER must do: 
ParkingRepository.createBasics(< number of required slot >)
val CONTROLLER = ControllerBuilder.createK(
	ParkingRepository.getUserRepository(),
	ParkingRepository.getParkingSlotRepository(),
	DoorQueues.getIndoorQueue(),
	DoorQueues.getOutdoorQueue(),
	LocalDoorState.get()
).get()
ParkManagerError.kt
A simple data class used for returning error after a method invocation. This class might not even be designed and exception could have been used instad, but Kotlin is a little bit difference from Java concerning exception (see Checked Expression). Then we decide to force the programmer to check if there was some error during the method invocation by an appropriate class for error. General, every controller method that can produce error returns an object Pair < T?, ParkManagerError? >, in which T is the type return ofthe method: then, if the error is not null, it means that during the method execution there was an error.

Token Generation

From the analysis nothing emerges about the token and his format: the consumer has specified nothing concerning this. However, from the analysis, is clear that the token is a unique string associated to the user after he has parked a car: the user must use this token in order to pickup his car.
The interface TokenGenerator defines the method generateToken() that can be used from the controller in his method assignTokenToUserAtIndoor() in order to generate the token. Because of the association between User and ParkingSlot is unique, the class SimpleTokenGenerator.kt is an implementation of the generator interface the use this relationship to produce surely univocal tokens. We also remeber that every generated token is associated to the user at the token property of the bean User and then saved into the data structures of the system thanks to an update invoked on the UserRepository.
